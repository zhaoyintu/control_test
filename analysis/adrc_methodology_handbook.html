温控 ADRC 整定方法论 —— 移植手册

把 AIC9 通道 1 上三天的摸索，沉淀成新设备上 1~2 天可执行的流程： 一套通用控制代码 + 一套通用工具链 + 每台设备半小时可获取的"参数包"。 本手册是路线图与验收关卡；具体操作细节引用两份流程文档执行。

00先判定：你的设备适不适用

这套方法论针对的对象类别：单输入单输出温控，执行器（加热器）的静态非线性可以测出来， 主导动态是"纯滞后 + 一阶惯性 + 积分/自衡"，扰动比控制带宽慢。逐条打勾：

  • ☐ 一个 MV（加热功率 %）控一个 PV（温度），控制周期 ≤ 主导时间常数的 1/10；
  • ☐ 加热器"给多少电出多少热"基本可重复（同 MV 同工况下马力稳定）；
  • ☐ 可以做开环 bump 测试（有手动模式、能一步写 MV）；
  • ☐ 负载/环境变化比阶跃响应慢（分钟级 vs 秒级）。
直接套用需要改造不适用
回流焊温区、注塑机筒加热、烘箱、热压板、退火/回火炉、实验炉——即本手册全流程 多温区强耦合（每区独立 ADRC + 耦合当扰动通常够；不够再上解耦）；带主动冷却的双向执行器（表做双段）；季节性大环境漂（锚点监控加密） 扰动快于可用带宽（强对流冲击、流量骤变主导）；相变负载主导；多变量强耦合必须整体优化的

01架构：为什么"特调"不可怕

方法论的核心是把机器差异压进数据层，代码与工具全厂一份：

层内容移植时
代码（通用）adrc_map_enabled.st：查表线性化 + 死区 FIFO + [7] 执行器惯性模型 lrTauF + [8] 记账温度修正 lrKqHot。无任何设备专用逻辑原样复用
工具链（通用）判读 theta_from_bump.py / fit_taue.py、回放辨识 replay_identify.py、孪生 twin.py、爬梯扫描原样复用
参数包（每台一份）q∞ 表、θ/τe、kq/tref、档位参数、安全表、锚点基线半小时标定获取
三句分工哲学（设计一切修正时的指南针）： 表管快非线性（执行器特性，标出来）；z2 管慢漂移（散热、负载、老化，在线估）； 标定管结构（滞后、惯性、温度依赖——z2 追不上的快速失配必须进模型）。 任何新修正必须回答一个问题：估错了往哪边错？——只接受"往慢、往不冲"错的设计 （范例：[8] 只修记账不修反查）。

04Step 1 · 摸底（用现有控制器，不改任何东西）

拿现有控制器（哪怕是烂 PID）跑 2~3 个阶跃 + 一段各温度点的稳态保持，存 10ms CSV。看五样：

  • 自衡还是积分型：恒定 MV 下温度收敛到平衡（自衡）还是持续爬（积分型）——决定 b0 的约定和孪生结构；
  • 滞后量级：阶跃后 PV 起动前的平台 ≈ θ 的数量级——决定 bump 保持时长设计（保持 ≥ 3~5×(θ+τe) 才能测出 q∞）；
  • 噪声 σ 与量化：稳态段 PV 的抖动——决定判读阈值和 wo 上限的量级；
  • 采样质量：时间戳间隔的中位/最大值——15ms 不规则可以忍，100ms 级则先修记录再标定；
  • 稳态 MV 曲线：各温度稳住需要百分之几——q 表低段的免费锚点 + 各温度平衡点（决定 bump 的 u₀ 预估）。
验收关卡 G1：能写出"对象类别 / θ 量级 / 噪声 σ / 采样 OK / 各温度 u₀ 表"五行摸底纪要。

05Step 2 · 标定 session（半小时真机，产出整个参数包的原料）

操作模板直接用两份流程文档改数字：theta_bump_test.html（θ + q 电平扫 + 锚点）、 taue_high_temp_test.html（高温区 τe，下跳为主）。五条核心纪律（每台设备都一样）：

  1. 稳温永远闭环：到位、稳住、回稳全部交给自动模式；手动只做"冻结 → 数 3~5 秒 → 一步跳变 → 按表计时 → 切回自动"（开环拿不住恒温，物理必然，别跟它较劲）；
  2. 越热越只做下跳：下跳只降温、零超温风险，且刹车问题的主角本来就是放热侧；
  3. 激励必须覆盖工作域——两个维度都要：功率段（低/中/高电平，尤其最高常用段）和温区（一直测到工作温度附近）。AIC9 两次翻车全在"数据边界外推"：高功率段一次、400°C 以上温区一次；
  4. 每拍前基线必须干净：上一拍没回稳就切手动的拍，判读脚本会自动作废——补拍比返工便宜；
  5. 收尾白捡锚点：结束前在工作温度闭环稳 60 秒（正常工况零风险），它是静态一致性与日后老化监控的基准。
python3 analysis/theta_from_bump.py 标定.csv     # θ、q 断点、稳态锚点、粘贴式 ST 变量表
python3 analysis/fit_taue.py 标定.csv            # 整段拟合 θ/τe/q∞, 按温度分组 τe(T)
验收关卡 G2：θ 样本极差合格；q 表单调、覆盖到最高常用电平； τe 各温区都有 ≥2 个有效下跳拍；工作温度锚点 ≥60s。缺哪拍补哪拍，不用整场重来。

06Step 3 · 数字孪生 + 盲测转正

孪生结构固定（twin.py，参数换新即可）：

MV → 纯滞后 θ → 静态表 q(u) × 高温缩水 g(u,T) → 一阶元件惯性 τe → 净加热 − 散热 h1·(T−c) → 积分 → PV
  • q 表填稳态马力 q∞（整段拟合），不要填短窗斜率——响应是"弯"的，短窗会把 τe 误算成马力不足（AIC9 高段被低估 40% 的教训）；
  • θ 与 τe 开环不可分（对光滑响应可互换），总滞后由闭环行为锚定——先按开环拟合填，闭环复现不上再调分配；
  • 转正标准（关卡 G3）：孪生复现一段没参与拟合的闭环数据（不同日期/不同参数最好）， 关键指标（上升/超调/settle/振铃周期）偏差 ≤20%。复现不了就回 Step 2 补数据，不要带着不合格的孪生进爬梯—— AIC9 的 v1 孪生缺 τe 结构，"仿真说 wc=8 没事"直接导致真机全场振荡。

07Step 4 · 仿真爬梯（标称 + 应力，出 2~3 档）

  • 在孪生上扫 wc/wo/wr/nd（模板 ladder_v2_scan.py），每个候选都要过应力角：增益 ±15%、滞后 +30%（数值按该设备的实测日漂调整）；
  • 输出 2~3 档：第 1 档求全应力零超调（验证代码和表没填错用）；第 2/3 档按性能目标（到达时间/超调预算）取舍；
  • 每一档必须写下预期数字（上升/超调/settle）——没有预期数字的上机测试无法判读；
  • 物理上限先算清楚：滑行 ≈ 进弯速度 × (θ+τe)，超调预算 ÷ 滑行系数 = 允许的进弯速度——"要多快"和"能多准"的取舍在仿真里谈妥，别到现场即兴发挥。
验收关卡 G4：档位表（参数 + 预期 + 应力最差值）成文，写进代码文件头。

08Step 5 · 上机爬梯与长期运维

1

爬梯纪律

一次只动一档/一个开关；每档先回归（新功能关闭态应复现旧行为）再开新功能； 跳档是事故的头号来源（AIC9：跳过第 2 档直接 wc=2/wr=4 → 12~15° 超调）。
2

每场测试后：回放辨识核对

python3 analysis/replay_identify.py 数据.csv 阶跃时刻 —— 用实测 PV 喂控制器方程、比对命令 MV，确认实跑参数 = 下发参数（RMSE <2% 坐实）。 "填的和跑的不一样"这类下发事故，靠它一分钟定案。
3

指标 vs 孪生对账

偏差 ≤20% → 爬下一档；>20% → 停，找差异来源（新机理？状态漂移？）， 回 Step 2/3 补标——AIC9 的 τe、高温缩水都是这样被"对账"逼出来的。
4

长期运维：把 z2 和锚点当仪表盘

生产上持续记录 z2 稳态值（≈当前散热负荷：开门是尖峰、放料是台阶、老化是慢漂）； 任何"稳住≥30s"的日志段自动出稳态锚点与表对比。判据：偏差 <10% 不动； 10~20% 复标 q 表（半小时）；>20% 或超调特性变化 → 连 τe 一起复标。换加热元件/大修必复标。
负载与工况的边界要心里有数：标定是"空炉/固定负载"快照。 放入大热容工件 = 环路增益被稀释（z2 兜底，稳但变慢）；传感器测的是炉温不是料温； 冷炉体首烧超调偏大属预期（冷壁吸热未建模）——正式评估用热机后的数据。

10十条元教训（比工具更值钱的部分）

  1. 标定质量的上限由实验设计决定，不由拟合算法决定——激励覆盖不到的域，模型在那里必然是猜的；
  2. "仿真说行、真机打脸"时，第一步是回放辨识排除"参数没下发"，第二步才怀疑模型结构；
  3. θ 和 τe 开环不可分；总滞后由闭环锚定。θ 这类"阶梯参数"最小二乘动不了，要网格扫；
  4. q 表必须填稳态马力 q∞；短窗斜率会把执行器惯性误记成"马力小"；
  5. MV 是零阶保持信号，重采样必须前值保持——线性插值会把阶跃劈碎导致漏检；
  6. 稳温是闭环的活，手动只做一步跳变；判读要漂移容忍（趋势线判据）；
  7. 一切修正设计成"估错落安全侧"（慢而不冲）；修记账优于修反查；
  8. ESO 的"积分"是 z2：冲刺期账实不符 → z2 记假账 → 刹车吐账 = 超调。看见反常超调先画 z2 轨迹；
  9. 没有预期数字的测试无法判读；没有验收关卡的流程必然跳档；
  10. 参数包（表+动态+档位+安全表）与代码分离，一台一份；锚点偏差是复标的闹钟。
