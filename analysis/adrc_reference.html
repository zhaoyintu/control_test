⚠ 全文所有对比数字都来自仿真，不是真实硬件测试结果——先说清楚模型是什么来路

动态部分（τ≈13s、死区L≈0.2s）是真实测出来的——来自脉冲测试的FOPDT拟合和真实闭环数据的死区实测。静态非线性曲线（功率-温度关系）不是从原始数据拟合出来的，是构造出来、只经过几个真实点标定的近似曲线：100℃≈2.6%MV、400℃≈21.5%MV 这两个真实闭环稳态点，加上 40% 功率点真实增益 K≈26.2 定的曲率——没有做过大范围、多功率点、每点保持足够久（≥5τ）的开环测试来真正描出这条曲线。

可信的是结构性结论：ESO 能自动吸收非线性和扰动、不需要精确模型，这是算法本身的性质，不依赖这条曲线画得准不准。不可信的是具体数字：1.16s、0.49% 这些只是在这条近似曲线上跑出来的量级参考，换一条更准的曲线，数字会变。要拿到能用于生产调参的真实数字，需要先做长时间保持功率点测试（每个功率点保持 ≥5τ，覆盖实际使用的整个功率范围）把这条曲线真正测出来，再重新跑一遍仿真——这一步目前还没做。

01

原理

不精确建模过程本身，而是实时估计并抵消"总扰动"。

ADRC（Active Disturbance Rejection Control，自抗扰控制）的核心部件是扩张状态观测器（ESO）：把执行器非线性、没建模的动态、外部扰动统一当成一个不需要知道具体成因的黑箱量 f(t)，实时估计、实时抵消。控制律只需要在抵消完 f(t) 之后、近似一个干净积分器的对象上做比例反馈。

SV → 参考整形(TD) → 控制律 → 对象；PV 与延迟后的 MV 反馈进 ESO，ESO 的估计再反馈回控制律

四个方程

对象模型 · Plant
y' = b0·u + f(t)
f(t) 吸收：真实增益与 b0 的偏差、未建模动态、外部扰动 —— 不需要知道 f(t) 具体是什么
参考整形 · TD (二阶临界阻尼)
v1' = v2 v2' = -2ωr·v2 - ωr²·(v1 - SV)
把阶跃 SV 变成平滑过渡轨迹 v1（及其速率 v2）—— 解耦"多快接近目标"和"闭环多快纠偏"
扩张状态观测器 · ESO
ż1 = z2 + b0·u(t-L) + 2ωo·(y - z1) ż2 = ωo²·(y - z1)
u(t-L)：喂给 ESO 的是延迟后的 MV，和对象实际感受到的一致 —— 忽略这点会导致真实的震荡（见 §02 设计笔记）
控制律
u0 = ωc·(v1 - z1) u = (u0 - z2) / b0
先对"干净"的状态估计做比例反馈，再减去估计出的总扰动

为什么这次比 MPC 更对症

  • b0 不需要精确 —— 执行器增益在识别出的功率区间变化约 2.5 倍，MPC 需要把这条非线性曲线显式辨识准，ADRC 的 ESO 会把估错的部分自动折算进总扰动吸收掉。
  • 不关心"为什么" —— 之前发现真实数据里同一稳态点在不同时刻要求的 MV 不完全一致（更可能是停留时间不够长，而非真的有第二个慢动态），ESO 不需要先搞清楚成因，照样吸收。
  • 调参旋钮从约 15 个压缩到 4 个 —— 现有 FB_AFPID_Base 里四档变速积分增益、detach 阈值、抗饱和上下限等经验参数，替换为 b0 / ωc / ωo / ωr 四个物理意义清晰的数。

代码实现

先在 Python 里验证，再落到 IEC 61131-3 Structured Text。

仿真验证用 adrc_step_linearized()

这是 §03 最终验证结果对应的版本——线性化过静态非线性、去掉了测试脚手架，只留核心递推。

adrc_core.py
def shape_ref(state, sv_raw, dt, wr):
    """Reference shaping -- critically damped 2nd-order tracking filter (the "TD").
    Decouples how fast we approach a new SV from how fast the loop rejects
    disturbance."""
    v1 = state.get('rv1', sv_raw)
    v2 = state.get('rv2', 0.0)
    v1n = v1 + dt * v2
    v2n = v2 + dt * (-2*wr*v2 - wr*wr*(v1 - sv_raw))
    return v1n, v2n


def adrc_step_linearized(state, y_meas, sv_raw, dt, wc, wo, wr, nd, y_ss, y_ss_inv, tau):
    """One control-loop scan of first-order ADRC, Hammerstein-linearized against
    the KNOWN static map g(u)=y_ss(u). Commanding in v=g(u) space makes the
    v->y path an exact unity-gain FOPDT, so b0=1/tau exactly -- nothing to
    guess. The ESO only corrects what's genuinely unknown (real disturbance,
    map error), not the actuator's own already-characterized nonlinearity.

    state persists {z1, z2, vbuf, rv1, rv2}. vbuf is a length-(nd+1) delay
    FIFO so the ESO sees the SAME delayed v the plant does -- vanilla ADRC
    has no other way to know about a dead time it isn't told about.
    """
    b0 = 1.0 / tau
    if 'z1' not in state:
        state = {'z1': y_meas, 'z2': 0.0, 'vbuf': [y_ss(0.0)] * (nd + 1)}
    rv1, rv2 = shape_ref(state, sv_raw, dt, wr)

    z1, z2, vbuf = state['z1'], state['z2'], state['vbuf']
    v_delayed = vbuf[-1]                      # committed nd steps ago -> acting now
    err_o = y_meas - z1                       # ESO innovation
    z1n = z1 + dt * (z2 + b0*v_delayed + 2*wo*err_o)
    z2n = z2 + dt * (wo*wo*err_o)

    u0 = wc * (rv1 - z1n)                     # feedback on the CLEAN estimate
    v_cmd = (u0 - z2n) / b0                   # cancel disturbance -> temperature-equivalent demand
    u_cmd = y_ss_inv(v_cmd)                   # invert the KNOWN static map -> real MV%

    u_sat = max(0.0, min(100.0, u_cmd))
    v_applied = y_ss(u_sat)                   # what v actually happened, post-clamp
    state = {'z1': z1n, 'z2': z2n, 'vbuf': [v_applied] + vbuf[:-1],
              'rv1': rv1, 'rv2': rv2}
    return u_sat, state

FB_ADRC_TempCtrl_SISO.st —— PLC 骨架

接口风格照抄 FB_AFPID_Base 的习惯（lr/x 前缀、E_FB_RUNST 状态机），方便直接对比、替换。不是可以直接烧录的产品代码——上机前需对照离线仿真验证一遍，这是任何新控制算法第一次上真实硬件都该做的事，不是这份骨架偷懒。

FB_ADRC_TempCtrl_SISO.st
FUNCTION_BLOCK FB_ADRC_TempCtrl_SISO
(*
    First-order linear ADRC (Active Disturbance Rejection Control), SISO,
    Hammerstein-linearized against the known static actuator map g(u).
    Proposed alternative to FB_AFPID_Base for the 100-400 degC band.

    Five tuning inputs total (lrTau, lrWc, lrWo, lrWr, plus the aMapU/aMapV
    breakpoint table) vs. FB_AFPID_Base's ~15 empirical knobs (four KIAnti*
    rates, detach thresholds, anti-windup limits, incomplete-diff filter
    constants). See the accompanying tuning guide for how to set each on real
    hardware -- bump-test procedure, order: tau/map -> Wo -> Wc -> Wr.

    WHY LINEARIZED: rather than picking one scalar lrB0 as a rough guess at
    K/tau and letting the ESO fight the gap between that guess and the true
    local gain wherever the plant happens to sit, this commands a temperature-
    equivalent v = g(u) and inverts u = g^-1(v) as the LAST step. The v->y path
    is then an exact unity-gain FOPDT by construction (dy/dt = (v-y)/tau), so
    b0 = 1/tau exactly -- nothing to guess. The ESO only has to correct what's
    genuinely unknown (real disturbances, map error), not the actuator's own
    already-characterized nonlinearity. Bench validation: this dropped true
    settling time on a 100-degree step from 6.46s to ~1.5s at matched (in fact
    smaller) overshoot, once combined with a correctly-identified dead time and
    a joint (Wc, Wo) search -- see adrc_vs_pid.py for the full numbers and the
    two mistakes (inflated dead time, one-parameter-at-a-time sweeps) that hid
    this region for several iterations.

    Dead time is compensated by feeding the ESO the SAME delayed v the plant
    actually sees (aVDelay FIFO) rather than the just-committed one -- vanilla
    ADRC has no other way to know about a dead time it isn't told about, and
    ignoring it caused real oscillation in bench simulation before this fix.

    NOT production code: validate against the offline reference simulation
    (same Python model used for the grid search) before trusting on hardware,
    and re-identify aMapU/aMapV from real long-hold pulse tests before use --
    the breakpoints here are illustrative, not the real furnace's curve.
*)
VAR_INPUT
    xEnable    : BOOL;
    xReset     : BOOL := FALSE;

    lrPV       : LREAL;    // measured PV, engineering units (degC)
    lrSV       : LREAL;    // setpoint, engineering units (degC)
    lrCycleT   : LREAL;    // execution period Ts (s)

    lrTau      : LREAL;    // dominant time constant (s) -- b0 = 1/lrTau, nothing else to guess
    lrWc       : LREAL;    // controller bandwidth (rad/s) -- sets closed-loop speed
    lrWo       : LREAL;    // observer bandwidth (rad/s), typ. 3-10x lrWc
    lrWr       : LREAL;    // reference-shaping bandwidth (rad/s)

    nDeadSteps : DINT;     // round(L / Ts) -- dead-time compensation for the ESO
    nMapPoints : DINT;     // breakpoints actually used in aMapU/aMapV
    aMapU      : ARRAY[0..15] OF LREAL;   // MV% breakpoints, strictly ascending
    aMapV      : ARRAY[0..15] OF LREAL;   // steady-state model output g(u) at each breakpoint
                                          // (from real long-hold pulse tests, not guessed)

    lrMVMin    : LREAL := 0.0;
    lrMVMax    : LREAL := 100.0;
END_VAR
VAR_OUTPUT
    xBusy  : BOOL;
    xError : BOOL;

    lrMV   : LREAL;   // controller output, 0-100 %
    lrZ1   : LREAL;   // ESO state estimate (~PV)               -- diagnostic
    lrZ2   : LREAL;   // ESO total-disturbance estimate         -- diagnostic
    lrRV1  : LREAL;   // shaped reference trajectory            -- diagnostic
END_VAR
VAR
    bFbState : E_FB_RUNST := E_FB_RUNST.unInit;

    aVDelay    : ARRAY[0..31] OF LREAL;   // FIFO of recent v=g(u) commands; [0] = newest
    lrB0       : LREAL;                   // = 1/lrTau, computed once at init
    lrRV2      : LREAL;                   // shaped reference's own rate (internal to TD)
    lrRV1Old   : LREAL;
    lrErrO     : LREAL;
    lrU0       : LREAL;
    lrVCmd     : LREAL;
    lrUCmd     : LREAL;
    lrVApplied : LREAL;
    lrVDelayed : LREAL;
    i : DINT;
END_VAR

//Init ADRC FB
IF xEnable THEN
    IF E_FB_RUNST.unInit = bFbState THEN
        lrB0 := 1.0 / lrTau;
        FOR i := 0 TO 31 DO
            aVDelay[i] := FC_InterpLookup(aMapU, aMapV, nMapPoints, 0.0);
        END_FOR
        lrZ1  := lrPV;
        lrZ2  := 0.0;
        lrRV1 := lrSV;
        lrRV2 := 0.0;
        bFbState := E_FB_RUNST.init;
    END_IF
ELSE
    bFbState := E_FB_RUNST.unInit;
    RETURN;
END_IF

IF xReset THEN
    bFbState := E_FB_RUNST.unInit;
    RETURN;
END_IF

IF E_FB_RUNST.run <> bFbState THEN
    xBusy := TRUE;
ELSE
    xBusy := FALSE;

    // === 1) reference shaping (critically damped 2nd order tracking filter) ===
    // Decouples "how fast we approach a new SV" from "how fast the loop rejects
    // disturbance". Tune lrWr against an explicit overshoot budget.
    lrRV1Old := lrRV1;
    lrRV1 := lrRV1 + lrCycleT * lrRV2;
    lrRV2 := lrRV2 + lrCycleT * (-2.0*lrWr*lrRV2 - lrWr*lrWr*(lrRV1Old - lrSV));

    // === 2) ESO: reconcile estimate with this scan's measurement ===
    // Uses the delayed v (nDeadSteps old), not the just-committed one -- this is what
    // "vanilla ADRC applied to a plant with known dead time" should actually do.
    lrVDelayed := aVDelay[nDeadSteps];
    lrErrO := lrPV - lrZ1;
    lrZ1 := lrZ1 + lrCycleT * (lrZ2 + lrB0*lrVDelayed + 2.0*lrWo*lrErrO);
    lrZ2 := lrZ2 + lrCycleT * (lrWo*lrWo*lrErrO);

    // === 3) control law: proportional feedback on the ESO's clean estimate,
    //        cancel the estimated total disturbance, invert the KNOWN static map ===
    lrU0   := lrWc * (lrRV1 - lrZ1);
    lrVCmd := (lrU0 - lrZ2) / lrB0;                                   // temperature-equivalent demand
    lrUCmd := FC_InterpInverse(aMapU, aMapV, nMapPoints, lrVCmd);     // -> real MV%
    lrMV   := LIMIT(lrMVMin, lrUCmd, lrMVMax);

    // === 4) advance the delay FIFO with what v ACTUALLY happened (post-clamp!) ===
    lrVApplied := FC_InterpLookup(aMapU, aMapV, nMapPoints, lrMV);
    FOR i := 31 TO 1 BY -1 DO
        aVDelay[i] := aVDelay[i-1];
    END_FOR
    aVDelay[0] := lrVApplied;
END_IF

IF E_FB_RUNST.init = bFbState THEN
    bFbState := E_FB_RUNST.run;
END_IF

(*
  FC_InterpLookup(aX, aY, n, x)   : monotonic breakpoint lookup, linear interpolation, y = g(x)
  FC_InterpInverse(aX, aY, n, y)  : same table, solved for x = g^-1(y) (aX,aY both ascending
                                    since g(u) is monotonic increasing)
  Both are ~15-line binary-search + lerp helpers -- same two helpers already needed for the
  MPC design (FB_MPC_TempCtrl_SISO.st), reusable as-is.
*)
⚠ 调试中踩到的坑
  • ESO 忽略死区会震荡 —— 一开始把上一拍的 MV 直接喂给 ESO，相当于假设零死区。真实死区 L≈0.5s，配合较快的 ωc，ESO 会对"还没来得及生效的信息"做反应，产生真实的震荡。修法：喂给 ESO 的是 nDeadSteps 步之前的 MV（上面 aUDelay 那个 FIFO），跟对象实际感受到的延迟对齐。
  • 参考轨迹的速度前馈用错了阶次 —— 曾经把 TD 的速度项 v2 直接加进控制律，这是二阶 ADRC（运动控制常用）的做法，搬到一阶温度对象上是错的：整形速度 ωr 越快，这项前馈反而制造出越大的冲击，超调从 0.5% 飙升到 56%。去掉之后超调稳定在 0.5% 左右，才能放心地把 ωr 调快。

调参决策流程 —— 拿到测试结果之后具体怎么做

下面这张图是整个调参过程的完整闭环：每次小阶跃测试之后，对照三个指标逐一检查，不达标就调对应的参数、回到第一步重新测试，直到三项同时达标，再做一次联合校验，才进入全量程测试和真实硬件上线。

小阶跃测试 → 记录四个数 → 逐项检查（不达标就调整、回到小阶跃重测） → 三项达标 → 联合校验 → 全量程 → 上线

1 · b0 —— 不算"调参"，是复用辨识结果

b0 ≈ K/τ。τ≈13s（脉冲辨识里稳定），K 在真实 300-400℃ 工作区间取 20-25（插值 20% 档 12.5 和 40% 档 26.2 之间）之间，起步 b0≈1.7。现场唯一验证：挑一个稳态点打个小阶跃，看 MV 最终稳定值是否落在合理范围（不顶到 0% 或 100%），不合理说明数量级错了，不需要精调。

✓ 验证过的改进：把已知的非线性提前线性化

调 ωo 之外，还有一条更根本的路：g(u)（执行器静态非线性曲线）本来就要靠长时间保持测试识别出来——干脆把这条已知曲线提前"拉直"。控制器不再直接命令 MV%，而是命令一个温度当量 v=g(u)，最后一步再用 g⁻¹(v) 反解成真实 MV%。这样 v 到 y 这条路径变成精确的单位增益 FOPDT，b0=1/τ 精确成立，不需要再猜一个"平均" K——ESO 只用纠正真正未知的部分，不用再花时间"发现"一个本来就知道的非线性。实测（同一个 300→400 阶跃）：

配置90%上升超调真正稳定(±1℃)
原方案（猜 b0，ωo=3×ωc）1.60s≈0%6.46s
线性化（精确 b0=1/τ，ωo=3×ωc）2.50s0%4.12s
线性化（精确 b0=1/τ，ωo=5×ωc）2.44s0.09%3.46s
真实系统（实测参考）~1.12s—3.47s

线性化 + ωo=5×ωc 这组真正稳定时间基本追平真实系统，超调依然可以忽略——但当时 90% 上升时间的代价还在（2.44s）。试过把 ωc 调回去想要回上升速度，超调和尾巴又跟着变差，一度以为这是躲不开的权衡。前提：g(u) 已经被扎实识别出来——直接复用前面建议的长时间保持功率点测试，不是额外的新工作。源码见 analysis/adrc_linearized.py。

✓✓ 那个权衡其实是两个自己挖的坑，不是物理限制

被问到"为什么不能三项同时好"之后重新查了两件事：① 前面所有仿真里死区一直用 L=0.5s——这是"0.19-0.61s 这个范围"里偏保守的取整，但 300-400℃ 这个区间实测死区specifically 只有 0.19-0.21s，用大一倍的死区，白白压低了可达到的 ωc 上限；② 之前全是一维扫描（一次只调一个参数），从没联合搜索过 (ωc, ωo)。把死区改对、配合线性化，对 (ωc, ωo) 做二维网格搜索，找到一大片 三项同时好 的区域——不需要在速度和稳定之间选边站：

4 · ωr —— 对着明确的超调预算调，不是"越小越好"

先跟团队定一个具体数字："超调不超过多少算合格"，再用同样的扫描方法找 ωr——含糊地说"越小越好"会无限牺牲速度，没有停止条件。

离线重放，不碰硬件
